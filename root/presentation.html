Requirements & Docs

Functional requirements & how docs mirror them

Functional requirements

What the server must do

  • Accept TCP connections, parse HTTP, send correct responses
  • Routing & static files; proper status codes (404, 405, …)
  • Many clients without blocking · keep-alive & clean close
  • Uploads, deletion, CGI, custom error pages
  • Handle invalid requests safely — never crash
Documentation structure

Mirrors the architecture

  • Core layer — sockets, event loop, connections
  • HTTP layer — parser, serializer, headers, cookies
  • Application layer — router, handlers, CGI, config
  • Config examples — uploads, redirects, error pages
  • Request flow & event-loop sections tie it together
Each requirement maps to a documented section — and to a handler in code.

Testing · Manual vs Automated

When to automate, when to do it by hand

Automated · stable logic

Run it on every commit

  • Request parsing & response serialization
  • Error paths & protocol edge cases
  • Config validation · CGI lifecycle
  • Documented inside test files; executed via cargo test
  • Catches regressions before they reach a PR
Manual · user-facing behaviour

What automation misses

  • Real browser behaviour — Chrome, Firefox, Safari
  • curl probes for headers, cookies, redirects
  • Upload & delete flows on the real filesystem
  • Custom error pages rendered visually
  • Documented in PR comments & issues with steps + expected result
Results are shared on GitHub PRs and Discord — bugs reproduced, discussed, fixed, then merged.

CI · Why it matters

What CI does for the codebase

01 · QUALITY GATE
Nothing reaches main unchecked
Build, lint, format, and the full test suite must pass for every push and PR. Regressions caught before review.
02 · FAST FEEDBACK
Minutes, not days
A complete pipeline run finishes in roughly four minutes. The author sees impact while context is fresh.
03 · REPRODUCIBILITY
The same checks for everyone
CI runs on a clean Macos environment with the stable toolchain.
04 · SAFETY NET
Confident refactoring
With every commit verified end-to-end, restructuring internals becomes routine instead of a risk.

CI · Pipeline

Pipeline stages — every push, every PR

01
CHECKOUT
Pull commit, set up stable Rust toolchain
02
CHECK
cargo check — fast type/borrow validation
03
CLIPPY
Lint with -D warnings — every warning fails
04
FMT
cargo fmt --check — uniform style
05
TEST
cargo test --include-ignored — full suite
06
COVERAGE
cargo tarpaulin → codecov upload
Each stage is fail-fast. A red signal stops the pipeline at the earliest possible point.

CI · Config & Results

The pipeline in YAML — and what it reports

.github/workflows/ci.yml
name: CI

on: [push, pull_request]

jobs:
  test:
    runs-on: macos-latest
    steps:
      - uses: actions/checkout@v3
      - uses: dtolnay/rust-toolchain@stable
      - run: cargo check
      - run: cargo clippy -- -D warnings
      - run: cargo fmt --check
      - run: cargo test --include-ignored
      - run: cargo tarpaulin --out Xml
      - uses: codecov/codecov-action@v3
